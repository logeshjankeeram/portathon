Detecting GNSS/GPS Spoofing for Port Security

This project demonstrates a practical, multi-method framework to detect maritime spoofing in real time. It includes a live events UI, a dedicated port authority dashboard, and an interactive dead-reckoning demo map.

Challenge – Port of Klaipėda

Recently, the Port of Klaipėda has encountered increasing cases of GPS signal interference. Currently, there is no alert system in place—neither for those participating in vessel traffic nor for the port’s cargo handling companies that depend on GPS signals for their operations. Such disruptions create serious risks for safe navigation, especially in zones with dense traffic of ships and tugboats and can disrupt the smooth execution of cargo handling tasks.

Interference of this kind may result in inaccurate navigation, hazardous maneuvers, or even incidents. Reliable positioning information is becoming ever more crucial for the safety and efficiency of port operations.

Challenge objective: Design a real-time GPS interference detection and warning system that will automatically notify vessels and cargo operation participants in case of signal disruptions. The system should:

  • Provide clear and tangible benefits to end-users by delivering timely alerts
  • Include a technological concept for how an innovative platform can enable quick identification of positioning issues
  • Reduce the risk of navigation errors
  • Help prevent potential incidents within the port area for all who rely on GPS-based equipment

Our Solution: MANA-based Spoofing Detection & Alert System

What We Built

We adapted the MANA framework (MAritime NMEA-based Anomaly detection) — a software-only method for detecting GNSS spoofing from regular NMEA-0183 data — and wrapped around it a live dashboard and actionable response protocol for the Klaipėda Port Authority.

  • Lightweight & retrofit-ready: MANA monitors existing NMEA strings and runs anomaly detection in software.
  • Modular detection engine: We combine multiple anomaly methods (residuals, consistency checks, deviation thresholds) so weaknesses in one method are mitigated by others. MDPI paper
  • Alert layer: Traffic-light UI (green/amber/red) for operators; amber/red signals potential spoofing to VTS.
  • Response orchestration: Protocols for cross-check via shore radar, secure VHF/HF contact, and pilotage escalation.

What Makes It Innovative

  1. Cost-effective retrofit: Software-first deployment on existing vessels or edge boxes (Raspberry Pi/NUC).
  2. Anomaly fusion / ensemble detection: Fuses trajectory consistency, NMEA residuals, and thresholds to reduce false alarms.
  3. Operationally actionable: Detection mapped to clear operator protocols; traffic-light UI aids decision-making under pressure.
  4. Scalable across fleets & ports: Modular and software-driven; supports many vessels feeding one command center.
  5. TRL 5 maturity: Validated in relevant environments (dataset + simulations); ready for pilot deployment.

How It Can Be Scaled / Extended

  • Edge device packaging for ship NMEA buses
  • Cloud aggregation to correlate multi-vessel spoofing and regional events
  • ML models that learn per-vessel normal behavior for proactive alerts
  • Fusion with AIS and shore radar to raise confidence
  • Collaborative detection networks across Baltic ports
